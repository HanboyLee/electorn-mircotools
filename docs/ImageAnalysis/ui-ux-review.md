# LLM 圖片分析 · UI/UX 審查與改版方向

> 設計依據：`ui-ux-pro-max` Skill（design-system + UX domain + React stack）  
> 現狀代碼：`src/pages/AnalyzeByImage/`  
> 產物：本審查文檔 + `design-system.md` + `ui-flow.mermaid` + **可瀏覽器打開的** `ui-prototype.html` + `implementation-plan.md`

---

## 1. 功能與用戶目標

| 項 | 內容 |
|----|------|
| 頁面名稱 | LLM 圖片分析 |
| 核心任務 | 批量選擇圖片 → 用 LLM 生成 **Title / Description / Keywords** → 導出 CSV（供後續元數據寫入等） |
| 主流程 | 確認 API →（可選）調整提示詞 → 選圖 → 分析 → 查看/導出 |
| 用戶 | 桌面端圖庫/素材整理用戶（Electron） |
| 技術棧約束 | **antd + styled-components**；OpenAI / OpenRouter；設置裡的 `analysisPrompt`、密鑰與模型 |

## 2. 現狀 UI 問題（對照 skill 規則）

| 嚴重度 | 問題 | 對應 UX 規則 | 現狀表現 |
|--------|------|--------------|----------|
| **高** | 就緒條件不透明 | Disabled state clarity | 「開始分析」常因 `!openaiApiKey` 禁用；選了 **OpenRouter** 時仍只看 OpenAI 密鑰，文案與邏輯不一致 |
| **高** | 批量進度不明 | Progress Indicators / Loading States | 僅全局 `Spin`「正在分析圖片…」，不知第幾張、成功幾張 |
| **高** | 主操作區與上傳區順序混亂 | Forms / Hierarchy | 先出「開始分析」再出上傳區；未選圖時主 CTA 與上傳關係弱 |
| **中** | 提示詞佔位過大 | Progressive disclosure | `PromptEditor` 常駐 Card，擠壓隊列與結果 |
| **中** | 上傳列表信息密度低 | Media / Feedback | `listType="text"`，無縮略圖、無單項狀態 |
| **中** | 結果難掃讀 | Feedback / Charts&Data 精神 | 每文件一張 Card + Descriptions，批量時滾動成本高 |
| **中** | 成功反饋弱 | Confirmation Messages | 分析結束無摘要（成功/失敗計數） |
| **中** | 清除操作無確認 | Destructive action | 「清除上傳圖片」一鍵清空結果 |
| **低** | API 警告只提 OpenAI | Error clarity | 未區分 Provider / 模型未選 |
| **低** | 控制台 `console.log` 噪音 | — | 調試日誌遺留（實現時清理） |

邏輯層一併注意（改 UI 時修復，不單改皮膚）：

1. 主按鈕 `disabled` 與 `handleAnalyze` 校驗應與 **當前 Provider** 對齊。  
2. 選新文件時清空 `results` 合理，但應提示用戶「重新選圖將清空分析結果」或保留已分析項策略（見 §5.4）。  
3. 導出 CSV 與「元數據寫入」的銜接可在結果區加一句次要引導（非強制跳轉）。

## 4. 改版信息架構

```
┌──────────────────────────────────────────────────────────────────┐
│ 頁眉：標題「LLM 圖片分析」+ 一句話說明                            │
│      Provider 狀態徽章（OpenAI / OpenRouter + 模型）             │
├──────────────────────────────────────────────────────────────────┤
│ [可折疊] 分析提示詞  ·  預覽一行 / 展開編輯保存                    │
├───────────────────────────────┬──────────────────────────────────┤
│ 左：圖片隊列（始終列表）        │ 右：就緒檢查                       │
│ 拖放/點擊上傳 · 限高列表        │ API ✓  提示詞 ✓  圖片數 ✓          │
│ 每行：小圖 · 文件名 · 狀態 · 移除│ 當前進度（分析中）                 │
├───────────────────────────────┴──────────────────────────────────┤
│ 結果區：摘要統計 + 表格/可展開行（標題/描述/關鍵詞/錯誤）          │
├──────────────────────────────────────────────────────────────────┤
│ 底部固定欄：[清除] [導出 CSV]              主 CTA「開始分析」     │
└──────────────────────────────────────────────────────────────────┘
```

### 內部狀態機（可不渲染為 Steps UI）

| 階段 | 完成條件 | 主 UI |
|------|----------|-------|
| 配置就緒 | 當前 Provider 密鑰有效（OpenRouter 還需模型） | 頂部徽章綠/黃；否則 Alert + 前往設置 |
| 隊列就緒 | 至少 1 張合法圖片 | 網格非空；CTA 可點 |
| 分析中 | `analyzing === true` | 進度條 + 項狀態；CTA loading；禁止清除/改隊列（或僅禁止刪正在分析項） |
| 結果 | 至少完成一輪 | 摘要 + 結果表；可導出；可對失敗項重試（可選二期） |

> **決策**：不展示頂部 Steps。與「元數據寫入」一致，用就緒檢查 + CTA 狀態表達可否分析。

## 5. 關鍵界面元素

### 5.1 Provider / API 狀態條

- 展示：`Provider` +（OpenRouter 時）`模型名` + 連通狀態（已配置 / 缺少密鑰 / 缺少模型）。  
- 未就緒：`Alert` warning + 主按鈕「前往設置」。  
- **就緒判斷**（必須與設置一致）：

| Provider | 就緒條件 |
|----------|----------|
| openai | `openaiApiKey` 存在且格式合理（可保留 `sk-` 提示，但勿過度嚴格阻擋實驗 key） |
| openrouter | `openrouterApiKey` + `selectedModel` |

### 5.2 提示詞（Progressive disclosure）

- 默認：**摺疊**，顯示一行摘要（截斷）+ 「展開編輯」。  
- 展開：TextArea + 保存 / 取消 / 恢復默認（沿用現有 Settings 持久化）。  
- 保存成功：就近 success Alert 或 message，3s 消失。  
- 分析進行中：只讀，不可改。  
- **硬約束（用戶確認）**：不修改 `initialSettings.analysisPrompt` 預設字串，不替換/簡化用戶已有提示詞內容；改版只動外殼交互。

### 5.3 圖片隊列

- 支持多選、`image/*`；拖放區（dashed）+ **始終列表**（不做網格 / 視圖切換）。  
- 每行：左側小縮略圖（約 40×40）+ 文件名（溢出省略 + tooltip）+ 狀態 Tag + 移除：  
  - `待分析` · `分析中` · `成功` · `失敗`  
- 移除單項；「清除全部」走二次確認。  
- 重複文件名：保持現有「跳過 + warning」，但用文件名集合提示更清晰。  
- 建議接受：jpg / jpeg / png / webp / gif（與 LLM 能力對齊，文案標明）。

#### 5.3.1 列表為唯一形態；大量圖片時的滾動與效能

**已採納：始終列表。** 不論 1 張還是 500 張，隊列 UI 形態一致，降低認知切換；用 **限高 + 虛擬列表 + 懶縮略圖** 扛大批量。

| 隊列數量 N | 展示形態 | 滾動 / 虛擬化 | 縮略圖策略 |
|------------|----------|---------------|------------|
| **任意 N** | **緊湊列表**（左 40px 圖 + 文件名 + 狀態 + 移除） | 隊列區 `max-height ≈ 280–360px`，**區內滾動** | 懶加載 Object URL，卸載 / 離屏 `revoke` |
| **N ≥ 50** | 同上 | **虛擬列表**（只渲染可見行 ± overscan） | 僅可見行持有 Object URL |
| **N ≥ 25** | 同上 + 頂欄搜尋 | 同上 | 同上 |

**隊列容器（所有 N 共用）**

```
┌─ 圖片隊列 · 共 N 張 ────────────────────────── [搜尋文件名…] ─┐
│ 拖放區（固定高度，不隨 N 增高）                                │
│ ┌─ 可滾動視窗 max-height 280~360px ─────────────────────────┐ │
│ │  列表行（虛擬列表：只畫可見項）                              │ │
│ │  [40px圖]  filename.jpg              待分析          ×    │ │
│ └───────────────────────────────────────────────────────────┘ │
│ 底欄：區內滾動查看更多 · 分析按「全部 N 張」執行                │
└────────────────────────────────────────────────────────────────┘
```

**交互補充**

| 能力 | 規則 |
|------|------|
| 視圖 | **無網格、無視圖切換**；固定列表 |
| 搜尋 / 過濾 | `N ≥ 25` 顯示搜尋框（按文件名）；過濾只影響**展示**，不從隊列刪除；主 CTA 仍處理全集（「僅分析篩選結果」為二期可選） |
| 移除 | 行尾 ×；可選多選批量刪除（二期） |
| 記憶體 | 禁止一次掛載全部大圖 blob；`N ≥ 50` 虛擬化 + lazy thumb 為硬約束 |
| 結果表 | 分析完成後表格：`N > 50` 虛擬滾動或分頁（每頁 50）；摘要卡始終只顯示統計數字 |
| 分析中 | 虛擬列表保留；當前分析行可 `scrollIntoView` 一次（避免每張都強滾） |

**刻意不做**

- 縮略圖網格 / 瀑布流相簿牆  
- 網格 ↔ 列表切換  
- 一期不做文件夾樹 / 磁碟目錄掃描（仍是用戶選文件進隊列）

### 5.4 選圖與結果策略

**推薦（一期）**：再次添加文件時 **合併隊列**（按 name+size+lastModified 去重），**不自動清空**已有結果；僅「清除」或「重新分析全部」清空結果。

若實現成本高，可暫留「替換 fileList 清空結果」，但須在 UI 上用 warning 文案明示。

### 5.5 進度

- 總進度：`Progress percent = completed / total`。  
- 文案：`正在分析 3/12 · filename.png`。  
- 單項：對應 Tag 切換為「分析中」→ 成功/失敗。  
- 禁止僅用一個大 Spin 佔滿結果區。

### 5.6 結果區

**摘要卡（分析結束後顯示）**：

| 指標 | 說明 |
|------|------|
| 合計 | 本輪處理張數 |
| 成功 | 有 `AnalysisResult` |
| 失敗 | 有 error |
| 待處理 | 未跑完時可顯示 |

**列表**：表格優先（antd `Table`）：

| 列 | 內容 |
|----|------|
| 預覽 | 小縮略圖 |
| 文件名 | text |
| 標題 | 可展開 |
| 描述 | 截斷 + 展開 |
| 關鍵詞 | Tag 列表 |
| 狀態 | 成功/失敗 |
| 錯誤 | 失敗時 message |

卡片堆疊可作窄寬降級，默認桌面用表。

### 5.7 底部操作欄

| 控件 | 行為 |
|------|------|
| 清除 | 確認後清空隊列 + 結果 + 錯誤 |
| 導出 CSV | 至少 1 條成功結果才可點；成功 toast |
| 開始分析 | 就緒且隊列非空；分析中 loading |

主 CTA 視覺權重最高（`type="primary"`），右對齊。

### 5.8 與下游銜接（輕量）

結果摘要旁次要文案：「可將 CSV 用於「元數據寫入」」—— **文字提示即可**，不強制自動跳轉。

## 6. 交互細節規格

| 場景 | 行為 |
|------|------|
| 未配置 API 點主 CTA | 攔截 + message + 可選自動 `navigate('/settings')` |
| 分析中點清除 | 禁用或 Modal「分析中無法清除」 |
| 單張失敗 | 不中斷後續；項上 error；摘要計入失敗 |
| 網絡斷開 | 若已有 NetworkStatus，可在頁頂提示「離線時無法分析」（沿用全局） |
| 導出無成功結果 | 按鈕 disabled + tooltip 說明 |
| 鍵盤 | Tab 順序：提示詞 → 上傳 → 清除 → 導出 → 開始分析 |

動效：狀態 Tag / Progress 150–200ms；**不要**結果列表 back.out 弹跳（skill：密集數據避免 overshoot）。

## 8. 非目標（本期不做）

- 不在本頁內嵌完整 API 密鑰表單（統一設置頁）  
- 不做多模型並發對比 A/B  
- 不做自動寫入 Exif（那是「元數據寫入」）  
- 不做雲端隊列/歷史記錄中心  
- 不引入新組件庫或 Skill 深色玻璃皮膚

## 9. 驗收標準（設計確認用）

1. 未配置 API 時，狀態與禁用原因一眼可見，且 OpenRouter 路徑正確。  
2. 選 ≥3 張圖時，隊列為**列表行**（小圖 + 狀態），非網格。  
3. 分析中可看到 n/total 與當前文件。  
4. 結束後有成功/失敗摘要，失敗項有錯誤信息。  
5. 提示詞默認不佔半屏。  
6. 底部欄主次操作清晰；清除有確認。  
7. 視覺與元數據寫入/設置同一 token 語言，無 teal 硬套。

## 10. 原型場景說明

見 `ui-prototype.html` 頂部場景切換：

1. **未配置 API** — Alert + CTA 禁用  
2. **就緒（少量）** — 隊列 4 張**列表**，主 CTA 可點  
3. **分析中** — 進度 2/4，列表分項狀態  
4. **結果** — 摘要 + 表；含 1 失敗  
5. **提示詞展開** — 編輯態  
6. **就緒 · 大量圖片** — 共 128 張：**始終列表** + 限高滾動 + 搜尋（虛擬列表示意）  

確認設計後再按 `implementation-plan.md` 改代碼。
